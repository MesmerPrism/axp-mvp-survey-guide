Values Through Stages

How values are computed and saved.

The important split happens after submission: one path keeps the current participant’s ASC factor scores in memory for immediate feedback, another writes durable MariaDB rows, and a later export path widens those stored rows into public and private bundles.

Stage overview

Stage Main datatype Why it matters
Definition load CSV / Google Sheet table, then R data.frame The system computes a definition hash from the loaded table and then filters by language.
Runtime questionnaire state R runtime metadata plus selected-language data.frame The running survey knows which instrument version and language were actually loaded.
Submission assembly R responses_df long table All responses are gathered before scoring and persistence.
ASC scoring R factor-score rows keyed by canonical scale_id Numeric values are mapped to the 11 ASC factors using the questionnaire’s embedded factor mapping.
Persistence MariaDB tables Raw collection is stored in normalized database tables.
Feedback and peer plotting In-memory self scores, cached peer context rows, and selected peer score query rows The participant sees a self polygon immediately, then optionally overlays peers chosen through canonical induction and dose filters.
Export Wide CSV bundles plus codebooks Public output removes free text and anonymizes identities; private output retains full detail.

Storage model

  • submission stores one survey session row with instrument, language, consent, hash, and duration metadata.
  • response_numeric stores one row per numeric item answer keyed by item_id.
  • response_text stores one row per text or context field; q0/q1 context persists even when full text storage is reduced.
  • score stores one row per canonical ASC factor score with scale_id and score_value.
  • peer_plot_submission_cache stores one row per submission with raw plus canonical induction/dose context and completeness flags.
Important distinction

The raw database is the source of truth for collection. The export bundles are the source of truth for distribution. The examples page shows both shapes side by side with mock files.

How the comparison selector and plot work

The immediate results view uses the current submission’s ASC factor score rows already in memory. Peer comparison values are loaded separately and only when the feedback step is active.

  • The participant’s own 11 ASC factor scores become the purple self polygon and points.
  • At submit time, db_sync_peer_plot_submission_cache() builds one cache row per submission from submission, the q0/q1 context stored in response_text, and the number of ASC factor rows written to score.
  • The comparison controls only show the recognized canonical bucket catalog back to the user: induction cannabis, psilocybin, alcohol, mdma; dose low, medium, high.
  • The active selection defaults to the current submission’s own induction and dose answers after those answers are canonicalized into peer tokens.
  • Once the participant chooses a pair such as psilocybin + medium, the app reads eligible submissions from peer_plot_submission_cache, joins to score on submission_id, filters to the selected canonical pair, and excludes the current submission.
  • Each returned peer factor score becomes an axis-aligned peer ring or density mark on the matching ASC axis, and the matching peers also contribute a median comparison polygon.
  • If too few peers exist in the selected bucket, the system omits the peer layer rather than fabricating comparison data.

What happens after submit

          
The clearest public distinction is between transient in-memory objects, normalized MariaDB tables, and the later public/private export files.

          

            

              
                

                  After submit
                  Immediate outcome
                

              
              
                

                  Validation passes
                  Responses are assembled into responses_df, then split into numeric and text/context rows.
                

                

                  ASC scoring runs
                  Canonical factor score rows become available for the current participant and later feed the self polygon.
                

                

                  DB write succeeds
                  submission, response_numeric, response_text, and score become durable, then peer_plot_submission_cache is rebuilt from the new submission context.
                

                

                  Feedback peer query runs
                  The app uses the selected canonical induction/dose pair to load only the matching peer score rows and render the peer overlay.
                

                

                  Later export
                  Normalized rows are widened and split into public or private CSV bundles plus codebooks.
