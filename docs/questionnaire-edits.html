How to adapt the questionnaire safely.

The questionnaire is managed as one stacked, multi-language table per version. The main discipline is keeping analytical identity stable where it should stay stable, and creating a new dated version when the questionnaire definition truly changes.

Mental model

  • One questionnaire version lives in one table.
  • Each language gets one block in the same columns.
  • Blank separator rows are formatting only and are removed by the loader.
  • item_id is the stable key used for runtime, scoring, and storage.
  • English rows define structure such as type, required status, order, and factor mapping.
  • Translation rows can stay structurally sparse when the English row already defines the contract.
Rule of thumb

Keep item_id stable when the question is still the same analytical construct. Create a new dated version tab when the questionnaire definition or interpretation materially changes.

Common change types

Change type Usually safe to keep the same item_id? Needs factor review? Typical action
Translation or wording cleanup Yes No Update the row text, then validate locally.
Required or active toggle Usually Sometimes Check navigation and submission behavior.
New substance or dose logic Usually No, unless scoring items changed too Review the override pairs and local walkthrough.
New scorable item or changed factor assignment Not automatically Yes Review scale_id, create a new dated version, and re-check downstream outputs.

Step-by-step adaptation path

  1. 1
    Classify the change.

    Decide whether you are changing wording, response logic, or the analytical meaning of the instrument.

  2. 2
    Edit the stacked questionnaire table.

    Keep one row per item_id per language, and keep all languages in the same tab.

  3. 3
    Check the structural columns.

    Make sure the authoritative English rows still carry the structural fields and any scorable rows still carry the right scale_id.

  4. 4
    Create or update a dated version tab.

    Use a new dated tab when the questionnaire definition changes in a way you may want to track separately.

  5. 5
    Validate locally.

    Walk the questionnaire, switch languages, test the substance and dose logic, and confirm the results view still completes.

  6. 6
    Promote and reload.

    Once the version is ready, reload the running survey so it resolves the updated Google Sheets version and computes a fresh definition hash.

What the collaborator should preserve

  • Keep one language block per language in the same table.
  • Keep structural truth on the English rows.
  • Keep item_id values stable unless there is a real analytical reason to change them.
  • Keep scorable rows mapped to the intended factor through scale_id.
  • Keep Google Sheets versioning at the tab level, not as ad hoc row metadata.
