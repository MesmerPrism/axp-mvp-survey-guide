What this guide covers

  • The questionnaire lives as one multi-language table per version.
  • Stable item_id values are the analytical backbone.
  • English rows define structure; translation rows mainly define wording.
  • Scorable items carry factor mapping through scale_id.
  • Numeric responses are scored, stored, reused for feedback, and later widened during export.
  • Public exports remove free text and anonymize submission identity.

Public boundary

This repo explains workflow and contracts. It does not include application source code, participant-level data, export files, or deployment secrets.

New collaborator path

1. Understand what counts as a version change

Not every wording edit creates a new analytical instrument. Some changes keep the same item_id; others require a new dated sheet tab and careful factor review.

2. Learn the Google Sheets workflow

The live questionnaire is maintained as a Google Sheet tab, but the real structure is a single stacked table with one language block per language.

3. Learn the value path

The same survey values move through normalization, scoring, MariaDB persistence, feedback generation, and export bundles. This guide shows that contract stage by stage.
